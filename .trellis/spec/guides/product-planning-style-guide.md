# 产品规划写法指南

> 用途: 写出能直接指导实现的 PRD, design 和 implement, 尤其适用于工具型桌面 app, UI 工作流和 MVP 规划.

## 什么时候使用

- 创建或修订 `prd.md`, `design.md`, `implement.md` 前使用.
- 任务涉及产品形态, UI 布局, 工具工作流或 MVP 范围时使用.
- 功能容易被误设计成控制台, 后台管理页, 营销页或普通 CRUD 页面时使用.

## 核心规则

规划文档默认使用中文. 只在必要技术名词, 命令名, 文件名, 组件名, API 名和包名中保留英文.

规划文档要描述真实产品体验, 不能只列功能清单.

## PRD 检查清单

- [ ] 先写清产品类型, 例如“桌面工具型 app”, “移动端表单工具”, “后台运营页面”.
- [ ] 写清这个产品不是什么, 尤其是已经出现过错误产品隐喻时.
- [ ] 用户价值要对应真实工作流, 不写抽象口号.
- [ ] 已确认事实, 推测和待确认问题分开写.
- [ ] 范围按 P0, P1, P2 和不做范围分层.
- [ ] P0 必须小到可以端到端实现和冒烟验证.
- [ ] 每个 P0 能力要说明为什么属于 P0.
- [ ] 好看但非核心的功能要降级, 不能让它抢占首屏.
- [ ] 验收标准必须是可测试的用户可见行为.
- [ ] 破坏性操作必须写清确认要求和影响范围.

## 设计文档检查清单

- [ ] 开头先写产品形态和设计原则.
- [ ] 如果已经出现过错误方向, 要明确拒绝错误布局或错误交互模型.
- [ ] 给出具体布局草图, 包含尺寸, 比例和顺序.
- [ ] 需要时写清默认窗口尺寸和最小响应式规则.
- [ ] 先定义全局外壳, 再定义单个组件.
- [ ] 每个可见组件都要写清:
  - 布局结构.
  - 默认态.
  - 加载态.
  - 禁用态.
  - 成功态.
  - 失败态.
  - 空状态或边界状态.
  - 主要交互.
  - 反馈位置.
- [ ] 写清反馈出现在哪里, 例如 toast, inline 状态或确认弹窗.
- [ ] 辅助文案和空状态要短, 要像工具产品, 不要像说明书.
- [ ] 前端职责和后端职责分开写.
- [ ] 每个跨层动作都要写数据流.
- [ ] 跨平台产品要写平台差异.
- [ ] 写实现约束, 防止后续走危险捷径.
- [ ] 视觉规范只写会影响实现的规则, 不写装饰性描述.

## 实施计划检查清单

- [ ] 按用户可见工作流组织阶段, 不按随意文件顺序组织.
- [ ] UI 形态错误时, 先改外壳和布局, 再接单个功能.
- [ ] 每个阶段下面都写验证方式, 不只在最后写验证.
- [ ] 写真实设备或真实平台的手工冒烟路径.
- [ ] 写清风险文件, 风险行为和回滚点.
- [ ] P1/P2 能力必须显式收口, 不能反过来定义 MVP.
- [ ] 未解决的产品决策放到最后的“继续实现前需要确认”.

## 工具型 app 规则

- [ ] 小工具产品优先使用工作台布局, 不默认使用侧边栏导航.
- [ ] 设备, 上下文和状态放在持久顶部区域.
- [ ] 高频动作放在首屏.
- [ ] 命令要转成人能理解的动作名, 例如“截图并打开”, “清当前应用数据”.
- [ ] 除非产品本身就是 shell 或控制台, 否则不暴露裸命令输入框.
- [ ] 列表视图默认是辅助能力, 除非用户的主工作流就是浏览列表.
- [ ] 破坏性动作要有明显视觉区分并要求确认.
- [ ] 工具卡片要紧凑, 易扫读, 风格一致.

## 错误写法和正确写法

### 错误

- “做一个 ADB GUI, 包含 Apps, Logcat, Screenshot, Text.”
- 因为功能有好几个, 就直接使用左侧侧边栏.
- 因为 app list 容易做, 就把 app list 放成默认首页.
- 只写“支持截图”, 不写文件保存到哪里, 是否打开, 有哪些状态.

### 正确

- “做一个 Android 开发测试桌面工具 app, P0 围绕截图保存, 文本传递, APK 安装, 快捷按键和当前应用动作.”
- 当控制台/侧边栏隐喻不适合产品时, 明确拒绝.
- 把高频工具工作台放到首屏.
- 截图行为写清: 保存路径, 文件名, 外部预览, 复制/显示文件入口, 禁用/加载/成功/失败状态.

## 来自 `06-26-adb-gui-app` 的可复用写法

后续规划要保留这些优点:

- PRD 先写清产品形态, 并明确它不是控制台或 app 列表管理器.
- PRD 将 app list 降级到 P1/P2, 因为它不是核心工作流.
- design 写出具体 960x640 桌面布局和 720px 最小宽度.
- design 写清双行顶部上下文栏, 卡片比例, 拖拽行为, toast 行为和卡片状态.
- design 对每个工具分开写前端职责和后端职责.
- implement 按用户工作流组织阶段, 每个阶段都有验证项.

当后续任务出现“重新规划需求”, “改设计”, “看看产品形态”, “沉淀写法”时, 先按这份指南检查.
